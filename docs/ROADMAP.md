# ROADMAP.md

> 6-месячный план до запуска real-money пилота. Разбит на 12 двухнедельных спринтов.
>
> Принципы планирования:
> - Каждый спринт имеет ОДИН главный deliverable. Если он не готов — спринт провален, разбираемся почему.
> - Следующий спринт не начинается, пока не закрыт предыдущий main deliverable. Можно перенести второстепенные задачи, но не основную.
> - Spike-задачи (исследования) — отдельно от продуктовых, не считаются за deliverable спринта.
>
> **Для Claude Code / AI-агента**: при работе над задачей всегда проверяй, в каком спринте мы сейчас, и не делай задачи из будущих спринтов "заранее" — это ломает приоритеты и съедает буфер.

## Спринт 1 (нед. 1–2): Foundation

**Main deliverable**: репозиторий, который запускается локально одной командой и деплоится в staging при пуше в `main`.

- Создать монорепо (pnpm workspaces или turborepo)
- Структура папок по `ARCHITECTURE.md` §2
- `docker-compose.yml` с postgres + redis для локалки
- Базовый NestJS skeleton в `apps/api`
- Базовый Next.js skeleton в `apps/web`
- GitHub Actions: lint + typecheck + test + build на PR
- Деплой в Railway/Render для api и web
- README с инструкцией "git clone → pnpm install → docker compose up → pnpm dev"
- Начальная схема БД (миграции пустые, но инфраструктура миграций работает)

**Definition of done**: коллега (или ты сам с другой машины) может клонировать репо и запустить локально за <10 минут. Push в main → через 5 минут staging обновлён.

## Спринт 2 (нед. 3–4): Auth + Users + Wallet

**Main deliverable**: пользователь регистрируется, видит профиль, видит виртуальный баланс с приветственным бонусом 1000 коинов.

- Регистрация email + пароль (Argon2id)
- Login + JWT (access + refresh)
- Refresh tokens в БД, отзыв
- Rate limiting на auth-эндпоинтах
- Профиль пользователя (имя, аватарка-инициалы, locale)
- Таблица `wallet_ledger` по `ARCHITECTURE.md` §5
- VIEW `wallet_balance` + endpoint `GET /me/wallet`
- Бонус 1000 виртуальных коинов при регистрации (через ledger, `bonus_grant`)
- Тесты: race condition на wallet (два параллельных списания не уведут в минус)
- Базовая страница регистрации/логина на фронте
- Защищённая страница "Profile" с балансом

**Definition of done**: можно зарегистрироваться, залогиниться, увидеть баланс 1000 коинов. Тест на двойное списание проходит зелёным.

## Спринт 3 (нед. 5–6): Game SDK

**Main deliverable**: формализован Game SDK интерфейс, написана "tic-tac-toe" как референсная игра-пример, прогоняется общий test suite для всех будущих игр.

- Реализован `packages/game-sdk` с типами из `ARCHITECTURE.md` §4
- Контрактный тест-сьют, который любая игра должна пройти (детерминированность, replay round-trip)
- Tic-tac-toe в `packages/games/tic-tac-toe` как reference implementation
- Документация: "как добавить новую игру" в `docs/ADDING_A_GAME.md`
- Игра НЕ интегрируется ещё в API — только пакет и тесты

**Definition of done**: `pnpm test packages/games/*` проходит. README "как добавить игру" — за 30 минут чтения новый разработчик может начать делать модуль.

## Спринт 4 (нед. 7–8): Первая игра — шашки + WS

**Main deliverable**: два пользователя могут сыграть партию в шашки в браузере через WebSocket, ход валидируется на сервере, победитель определяется сервером.

- `packages/games/checkers` на базе `draughts.js` (или собственная имплементация, если библиотека не подходит)
- Все методы Game SDK реализованы и покрыты тестами
- WebSocket-сервер (`apps/ws`) поднят отдельным процессом
- Match runtime в API: создание match, помещение state в Redis, обработка submit_move
- Frontend: компонент шахматной доски (или библиотечный, например `react-checkers` или собственный canvas)
- Можно вручную через админ-консоль создать match между двумя test-юзерами и сыграть
- Логика timeout-ов на ход

**Definition of done**: два браузера (можно incognito-режимы) соединяются к одному match-id, играют партию, сервер определяет победителя.

⚠️ На этом этапе ставок ещё нет — играем "вхолостую" чтобы изолировать сложность.

## Спринт 5 (нед. 9–10): Matchmaking + Elo + bet lifecycle

**Main deliverable**: пользователь нажимает "Играть в шашки за 100 коинов", через очередь матчится с другим, ставки списываются с обоих, победитель получает 180 (10 коинов rake).

- Matchmaking-сервис с Redis-очередями: ключ = `{game}:{stake}`
- Elo на пользователя по каждой игре (стартовый 1200)
- Полный bet lifecycle: `bet_lock` → `bet_settle`/`bet_unlock`, rake-запись
- API эндпоинт "встать в очередь" + cancel
- UI: кнопка "Играть", статус "Ищем соперника...", переход в комнату при match
- Обработка abandonment: если игрок дисконнектится >60 сек → tech defeat
- Интеграционный тест: симулируем 10 матчей подряд, проверяем что баланс платформы (rake) сходится с разницей в балансах игроков

**Definition of done**: е2е тест полного цикла зелёный. На стейджинге два юзера могут сыграть 5 партий подряд, балансы сходятся до копейки.

## Спринт 6 (нед. 11–12): Лобби, профиль, история, polish

**Main deliverable**: продукт визуально и UX-ово готов к показу beta-тестерам без объяснений "тут пока заглушка".

- Лобби: список доступных ставок, "Quick play", "Найти матч с друзьями" (по нику)
- Расширенный профиль: статистика побед/поражений, Elo по играм, последние 20 матчей
- Replay-вьюер: кнопка "Посмотреть" на каждом завершённом матче
- Push-уведомления через Web Push API (нашли соперника, началась партия)
- i18n каркас: RU как основной, EN-заглушки
- Полный mobile-first UI, протестирован на iOS Safari и Android Chrome
- Лендинг: главная страница "Что такое, как играть, кнопка регистрации"
- Terms of Service + Privacy Policy (шаблонные)

**Definition of done**: открываешь сайт с телефона незарегистрированным пользователем — за 3 минуты прошёл регистрацию, увидел лобби, понял что делать, сыграл партию.

## Спринт 7 (нед. 13–14): Турниры

**Main deliverable**: запускается турнир на 8 человек, entry 100 коинов, призовой пул 800, бракет-сетка, победитель получает 500.

- Модель турнира: bracket (single elimination на старте), entry fee, payout distribution
- Lobby турниров: список запланированных, регистрация
- Автостарт при наборе участников ИЛИ по расписанию
- Жизненный цикл: registration → live → finished
- UI бракета (визуализация)
- Уведомления "твой матч через 1 мин" / "ты вышел в полуфинал"
- Лидерборд турнирных очков (отдельно от Elo)

**Definition of done**: на стейджинге проводится турнир на 8 ботов (или live-тестеров), сетка раскручивается, призы распределяются корректно.

## Спринт 8 (нед. 15–16): Retention & Базовый антифрод

**Main deliverable**: новый пользователь по чужой ссылке получает бонусы, запущены лидерборды и базовые защиты от абуза (IP, Fingerprint) до старта беты.

- Реферальные коды (генерируются на user)
- Атрибуция при регистрации (cookie + параметр URL)
- Бонусные выплаты через ledger (`bonus_grant`)
- Daily login bonus: 50 коинов в день при заходе
- Серия побед: каждые 5 побед подряд — бонус 100 коинов
- Сезонный лидерборд (1 месяц), топ-10 получают коины в конце сезона
- Базовый антифрод: Device fingerprint (FingerprintJS) и IP-ASN lookup при логине
- Email-уведомления: "вы получили бонус", "новый сезон стартовал"

**Definition of done**: реферальный flow работает end-to-end, бонусы корректно начисляются. Запущен первый сезон лидерборда.

## Спринт 9 (нед. 17–18): Closed beta

**Main deliverable**: 50–200 живых пользователей, ежедневная аналитика, бэклог багов из реального использования.

- Подключить Posthog: ключевые события (registration, first_match, deposit, win, withdrawal_attempt)
- Дашборд метрик: DAU, MAU, D1/D7 retention, средн. матчей на юзера
- Set up Sentry с алертами на критичные ошибки
- Закрытая регистрация по invite-кодам
- Запустить в Telegram-канале/чате по настольным играм (50–100 приглашений)
- Daily standup с самим собой: смотришь метрики и багрепорты, фиксишь
- Опросник для beta-тестеров после 1-й недели использования

**Definition of done**: 30+ DAU стабильно в течение 2 недель. Собраны метрики, понятны проблемы. Не упало ни разу > 5 минут.

## Спринт 10 (нед. 19–20): Расширенный антифрод + админка

**Main deliverable**: реализованы продвинутые антифрод-правила (включая детект движков для шашек), есть админка для ручного review флагов.

- Async worker (BullMQ или встроенная очередь NestJS) для антифрод-проверок
- Анализ совпадений ходов с идеальной линией (Engine-perfect play) для шашек (KingsRow/Aurora)
- Правила Pair-collusion, New-account drain, Robot timing
- Admin UI: список флагов, детали матча, кнопки "Игнорить" / "Бан" / "Заморозка вывода"
- Audit log всех админских действий (отдельная таблица)
- Тесты на правила: smoke + по 2-3 кейса на каждое правило

**Definition of done**: можно симулировать каждый из 5 кейсов антифрода и убедиться, что флаг создан и виден в админке.

## Спринт 11 (нед. 21–22): Подготовка к real money

**Main deliverable**: можно через крипту пополнить кошелёк и вывести с него на тестовом стейджинге (через SaaS-провайдера для MVP).

- Интеграция с SaaS-провайдером крипто-платежей (например, NowPayments) для USDT TRC-20 (поднятие своей ноды TRON отложено для экономии времени)
- Воркер мониторинга входящих транзакций → ledger
- Withdrawal flow: запрос → ручное подтверждение в админке на старте → вызов API провайдера
- Лимиты: мин депозит $5, мин вывод $10, макс вывод в день $500 на старте
- Двойная подпись на все админские withdrawal-операции (две учётки админов)
- Responsible gaming flow: self-exclusion, лимиты депозитов в день

**Definition of done**: на стейджинге через настоящий USDT TRC-20 можно депнуть, сыграть, вывести. Все операции в ledger корректны.

## Спринт 12 (нед. 23–24): Real money pilot

**Main deliverable**: 20–50 человек играют на реальные деньги, обработано ≥10 успешных циклов deposit→play→withdraw без потерь.

- Запуск real money режима для invited only (отдельный whitelist)
- Параллельный UI-режим: пользователь может играть на коины ИЛИ на USDT
- Минимальная ставка $0.5, максимальная на старте $20 (для контроля рисков)
- 24/7 мониторинг: Sentry-алерты + Telegram-бот шлёт уведомления на любые антифрод-флаги
- Ручная проверка КАЖДОГО вывода первые 2 недели
- Daily финансовый reconcile: сумма USDT на кошельках = сумма по ledger
- Документ "incident playbook": что делать если двойной вывод, спорный матч, жалоба

**Definition of done**: 30 дней после запуска без финансовых инцидентов, обработано ≥10 выводов, есть хотя бы 5 повторных депозитов от одних и тех же пользователей.

## Что НЕ в этом roadmap (отложено за 6 мес)

- Нативные мобильные приложения
- Вторая игра (нарды) — спринты 13–14 уже в "post-MVP"
- Лицензирование
- Подписки VIP
- Стриминг матчей
- Чат между игроками (отключён по умолчанию)
- Соцсети (друзья, кланы)
- Маркетинг-кампании выше базового реферального уровня

## Буфер и риски

- Каждый спринт по факту планируй на 80% времени, 20% — на непредвиденное. Если выходишь за 100% второй раз подряд — режь scope, не сроки.
- Самые рискованные спринты: 4 (первая игра + WS), 5 (matchmaking + bet lifecycle), 11 (USDT). На них держи буфер +1 неделю в плане.
- Если в спринте 9 (beta) метрики провальные (D7 < 10%) — ОСТАНОВИСЬ. Не делай спринты 10–12. Pivot.

## Правила обновления roadmap

- Раз в спринт ретроспектива: что взял в работу, что сделал, что не сделал и почему.
- Если main deliverable не закрыт — он переходит в следующий спринт, и что-то из следующего вылетает.
- Изменения сроков фиксируются в этом файле с датой и причиной.
